# Releasing

Releases are cut by pushing a tag (`git tag v0.X.Y && git push origin v0.X.Y`).
GitHub Actions then builds, ad-hoc signs, and publishes three assets:

| Asset | Purpose |
|-------|---------|
| `ClaudePulse-X.Y.Z.dmg` | What users download — drag-&-drop install |
| `ClaudePulse-X.Y.Z.zip` | Sparkle auto-update archive |
| `appcast.xml` | Sparkle feed, EdDSA-signed |

It also bumps the Homebrew cask in the `homebrew-tap` repo (when configured).

## Why no notarization

Removing the Gatekeeper "Apple could not verify…" dialog would require Developer
ID signing + notarization, which is inseparable from a paid
[Apple Developer Program](https://developer.apple.com/programs/) membership
($99/year) — ruled out for this project. There is **no free alternative**:
Developer ID certificates only exist inside paid accounts, Apple's notary service
only accepts submissions from them, third-party "sign with our cert" services
violate Apple's terms and get their certificates revoked (which would kill every
installed copy at once), and fee waivers cover only nonprofits, schools, and
government entities. OSS signing services like SignPath are Windows-only.

The trade-off: macOS Gatekeeper asks for a one-time confirmation on first launch
of a downloaded copy (System Settings → Privacy & Security → Open Anyway, or
`xattr -dr com.apple.quarantine` — right-click → Open no longer works on
macOS 15+). Homebrew users avoid even that with `--no-quarantine`. Sparkle
auto-updates are unaffected — they verify the EdDSA signature, not an Apple
ticket, so updates never re-trigger Gatekeeper.

To switch to the notarized flow if a membership ever materializes: re-add a
Developer ID build (`CODE_SIGN_IDENTITY="Developer ID Application"`,
`ENABLE_HARDENED_RUNTIME=YES`, `--timestamp`) plus `xcrun notarytool submit …
--wait` and `xcrun stapler staple` for both the app and the DMG, with the
`MACOS_CERT_P12` / `MACOS_CERT_PASSWORD` / `APPLE_ID` / `APPLE_TEAM_ID` /
`APPLE_APP_SPECIFIC_PASSWORD` secrets. (The git history of this file and
`release.yml` contains the complete working version.)

## One-time setup: GitHub secrets

### `SPARKLE_ED_PRIVATE_KEY` (required)

The EdDSA key matching `SUPublicEDKey` in `project.yml`, generated by Sparkle's
`generate_keys`. Signs every appcast entry so installed clients accept the
update. Set via `gh secret set SPARKLE_ED_PRIVATE_KEY` or *Settings → Secrets
and variables → Actions*.

### `HOMEBREW_TAP_TOKEN` (optional — enables the cask bump)

Lets the release workflow push an updated cask to your tap. Without it, the
"Bump Homebrew cask" step is skipped and the DMG/zip/appcast still publish.

1. Create a public repo named **`homebrew-tap`** under your account
   (`github.com/psalkowski/homebrew-tap`). A `Casks/` folder is created on the
   first release.
2. Create a fine-grained Personal Access Token (*Settings → Developer settings →
   Personal access tokens → Fine-grained tokens*): under **Repository access**
   pick **"Only select repositories"** → `homebrew-tap` (the default "Public
   repositories" mode is read-only and hides all write permissions), then under
   **Permissions → Repository permissions** set **Contents** to **"Read and
   write"**. The auto-added *Metadata: Read-only* is expected.
3. `gh secret set HOMEBREW_TAP_TOKEN` in the `claude-pulse` repo (paste the
   token).

Users then install with:

```sh
brew install --cask psalkowski/tap/claude-pulse
xattr -dr com.apple.quarantine /Applications/ClaudePulse.app
```

Homebrew always quarantines casks. As of Homebrew 6 the `--no-quarantine` CLI
flag was removed **and** `HOMEBREW_CASK_OPTS=--no-quarantine` is ignored by
`brew install --cask` (the install command no longer threads the quarantine
option through — verified in `cmd/install.rb`), so an ad-hoc app always lands
quarantined. Clearing the flag with `xattr` (printed by the cask's caveats) is
the reliable bypass; *Open Anyway* in System Settings works too.

The canonical cask lives in [`Casks/claude-pulse.rb`](../Casks/claude-pulse.rb)
(the `version`/`sha256` placeholders are seeds — CI patches them per release).

## How the pipeline signs

- The app is built with the `project.yml` defaults (`CODE_SIGN_IDENTITY="-"`,
  hardened runtime off) and then re-signed ad-hoc with
  `codesign --force --deep --sign -` so the embedded widget is covered.
- The DMG is built unsigned (ad-hoc apps don't gain anything from a signed
  container); Gatekeeper evaluates the app inside on first launch.
- Local builds (`make run`, `install.sh`) are ad-hoc too — same as the release.

## Local DMG preview

`make dmg` (needs `brew install create-dmg`) builds `dist/ClaudePulse-dev.dmg`
with the same drag-&-drop layout — for checking the window layout without
cutting a release.
